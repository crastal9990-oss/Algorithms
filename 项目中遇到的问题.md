# 文件下载

## 传统方式

### 1. 动态创建 `<a>` 标签（最常用）

这是前端处理 Blob 数据的标准做法：

```js
function traditionalDownload(url, filename) {
  // 创建隐藏的 <a> 标签
  const a = document.createElement('a')
  a.style.display = 'none'
  a.href = url
  a.download = filename
  
  // 添加到 body 并触发点击
  document.body.appendChild(a)
  a.click()
  
  // 清理
  document.body.removeChild(a)
}
```

------

### 2. 配合 Blob 使用的完整流程

这是处理动态数据的常见写法：

```js
async function downloadWithBlob(url, filename) {
  // 步骤1：把整个文件下载到内存
  const response = await fetch(url)
  const blob = await response.blob()  // ⚠️ 关键：整个文件载入内存
  
  // 步骤2：创建一个临时的本地 URL
  const blobUrl = URL.createObjectURL(blob)
  
  // 步骤3：用 <a> 标签下载
  const a = document.createElement('a')
  a.href = blobUrl
  a.download = filename
  a.click()
  
  // 步骤4：清理内存
  URL.revokeObjectURL(blobUrl)
}
```

### 3.传统方式的变种（都很类似）

**方式A：直接使用 Blob URL**

```js
const blob = new Blob([data])  // data 可以是 ArrayBuffer、字符串等
const url = URL.createObjectURL(blob)
window.open(url)  // 或 location.href = url
```

**方式B：使用 `msSaveBlob`（IE 专用）**

```js
if (window.navigator.msSaveBlob) {
  window.navigator.msSaveBlob(blob, filename)  // IE 的传统方式
}
```

**方式C：使用 `FileReader`**

```js
const reader = new FileReader()
reader.onload = (e) => {
  const a = document.createElement('a')
  a.href = e.target.result
  a.download = filename
  a.click()
}
reader.readAsDataURL(blob)  // 转换为 base64，内存占用更大
```

## 传统方式的内存问题图解

假设要下载一个 **1GB** 的文件：

```
传统方式（Blob）：
网络数据 → [内存：1GB 完全存储] → 磁盘
          ↑
       内存爆炸！页面可能崩溃

流式方式：
网络数据 → [内存：1MB 缓冲区] → 磁盘
          ↑                ↑
       收到1MB立刻写入   写入后释放内存
```

**缺点**

- ❌ 内存爆炸：整个文件必须完全加载到内存

- ❌ 无法显示进度：下载过程用户看不到进度

- ❌ 无法暂停/续传

- ❌ 大文件会导致页面卡死或崩溃

- ❌ 移动端容易触发 OOM（内存不足）

**各浏览器的实际内存限制：**

- Chrome：单个标签页约 2GB（64位），但大 Blob 会导致严重卡顿

- Firefox：约 1.5GB，超过可能崩溃

- Safari：约 1GB，iOS 上更少（几百 MB 就可能崩溃）

## 我在项目中使用的方式是传统Blob下载方式

后端是**先打包完再返回blob**

```js
    async downLoad(id) {
      try {
        const res = await batchDownloadFiles({ ids: id }) 
        if (res.data) {
          if (res.data.success !== undefined && !res.data.success) {
            return this.$message.error(res.data.msg || '批量下载失败');
          }
          //整个文件载入内存：res.data 已经是完整的blob响应数据
          const url = window.URL.createObjectURL(res.data); //生成临时URL：createObjectURL() 创建内存中的本地链接
          const link = document.createElement('a');
          link.href = url;
          link.download = '下载文件.zip';
          document.body.appendChild(link);
          link.click(); // 模拟点击下载：通过 <a> 标签触发下载（从内存写到磁盘）
          document.body.removeChild(link);
          window.URL.revokeObjectURL(url); // 清理内存：revokeObjectURL() 释放内存
          this.$message.success('下载成功');
          this.selectedIds = [];
          if (this.$refs.multipleTable) {
            this.$refs.multipleTable.clearSelection();
          }
        }
      } catch (error) {
        this.$message.error('下载失败');
      }
    },
```

详细的状态变化

```
[时刻1] 发送请求
  ↓
  ✅ 状态: Pending（请求已发出，等待服务器响应）
  浏览器操作: 建立 TCP 连接、发送 HTTP 请求头

[时刻2] 服务器收到请求，开始处理
  ↓
  ✅ 状态: 仍然是 Pending
  浏览器操作: 等待服务器响应

[时刻3] 服务器返回响应头（HTTP Status: 200）
  ↓
  ✅ 状态: 从 Pending → 变为 200（但数据还在传输中！）
  浏览器操作: 
    - 解析响应头（Content-Type, Content-Length 等）
    - 准备接收响应体（body）

[时刻4] 开始接收响应体（文件数据）
  ↓
  ✅ 状态: 200（正在接收数据）
  浏览器操作:
    - 接收数据块1 → 放入内存缓冲区
    - 接收数据块2 → 放入内存缓冲区
    - 接收数据块3 → 放入内存缓冲区
    - ... 持续接收直到全部完成

[时刻5] 所有数据接收完毕
  ↓
  ✅ 状态: 仍然是 200（但数据传输完成）
  浏览器操作:
    - 通知 axios: "数据都收完了"
    - axios 解析数据（如果需要）
    - 触发 Promise resolve
    - 你的 await 结束

[时刻6] 你的代码继续执行
  ↓
  console.log('请求完成')
  创建 Blob、触发下载...
```

文件过大时，因为需要等全部数据传完（从网络传输到 JavaScript 内存），**长时间停留在时刻4正常接收数据**，用户点击下载后没有看到任何反馈

## 大文件下载

### 1. File System Access API流式下载

File System Access API 是浏览器提供的一套 API，允许网页**直接读写用户本地文件**。对于下载场景，它可以做到：

- ✅ 内存占用极低（几十 MB 的文件也只占几十 KB 内存）
- ✅ 支持任意大小文件（甚至 TB 级）
- ✅ 有原生进度反馈
- ✅ 用户可以自定义文件名和保存位置

**后端：立即返回响应头，不能等打包完，然后边打包边发送数据**

```js
// 前端配合流式后端
async downLoad(id) {
    // ========== 第1步：发起请求，获取响应流 ==========
  // fetch 是浏览器原生API，用于发送HTTP请求 ，fetch 的 await 只等响应头，不等数据
  // 与 axios 不同，fetch 返回的 response 对象可以访问原始数据流
  const response = await fetch('/api/aos-cloud-flight-operation/resultdata/batchDownLoad', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ ids: id })
  })
  
  // ⭐ 关键点1：此时 response 对象已经收到，但数据还在传输中
  // ⭐ 关键点2：response.body 是一个 ReadableStream（可读流），可以边接收边处理
  // ⭐ 关键点3：这一行代码没有等待数据全部传完，只是等待服务器响应头（状态码等）
  
  
  // ========== 第2步：弹出系统保存对话框 ==========
  // showSaveFilePicker 是浏览器原生API
  const fileHandle = await window.showSaveFilePicker({
    suggestedName: '下载文件.zip'
  })
  
  // ========== 第3步：创建可写流 ==========
  // createWritable() 会创建一个指向用户选中文件的写入流
  // 往这个流里写数据，就会直接写入磁盘
  const writable = await fileHandle.createWritable()
  
  // ========== 第4步：管道传输（核心！） ==========
  // response.body 是 ReadableStream（可读流，来自网络）
  // writable 是 WritableStream（可写流，指向磁盘文件）
  // pipeTo() 方法把可读流的数据自动传输到可写流
  await response.body.pipeTo(writable)  // 边收边写
    
  // ⭐ 关键点8：pipeTo 做了这些事情：
  //    1. 从网络读取一小块数据（比如 64KB）
  //    2. 立即写入磁盘
  //    3. 释放内存中的这一小块数据
  //    4. 重复 1-3 直到所有数据传输完成
  // ⭐ 关键点9：整个过程内存占用始终只有一小块（几十KB）
  // ⭐ 关键点10：数据不经过 JavaScript 内存，直接从网络流到磁盘
  
    
  // ========== 第5步：完成 ==========
  // pipeTo 完成后，writable 会自动关闭
  // 文件已经完整保存到用户选择的位置
  this.$message.success('下载完成')
}
```

**为什么内存占用低？**

传统方式是把整个文件先装进内存（`blob()` 或 `arrayBuffer()`），再触发下载。而流式写入的方式是：

- 网络收到 1KB 数据 → 立即写入磁盘 1KB → 内存释放这 1KB
- 整个过程内存里只保留一个"缓冲区"（通常几十 KB 到几 MB）

**缺点**

- ❌ Chrome/Edge 86+ 才支持，Firefox 和 Safari 不支持
- ❌ 必须在 HTTPS 或 localhost 环境下运行
- ❌ 需要用户交互触发（不能自动下载）

### 2. StreamSaver.js

**核心原理**

StreamSaver 解决的痛点是：**在不支持 File System Access API 的浏览器中，如何实现流式写入？**

它的实现方式非常巧妙，利用了 **Service Worker** 技术：

1. **注册 Service Worker**：StreamSaver 会注册一个 Service Worker，这个 SW 可以拦截网络请求
2. **创建"假文件"**：调用 `streamSaver.createWriteStream(filename)` 时，它创建了一个**特殊的下载链接**（`blob:http://...` 或 `http://localhost:...`）
3. **模拟写入**：当你向这个流写入数据时，数据实际上是通过 `postMessage` 发送给 Service Worker
4. **Service Worker 处理**：SW 收到数据后，通过 `fetch` 事件模拟一个文件响应，触发浏览器的下载机制，fetch(url).then(res => res.body)——从服务器拿数据流
5. **分块写入**：数据被分成小块发送，浏览器像接收普通 HTTP 响应一样，边接收边写入磁盘

**关键配置：mitm.html**

StreamSaver 需要配置一个 `mitm.html`（中间人页面），这是它的工作机制决定的。

```js
import streamSaver from 'streamsaver'

// 方法1：使用 CDN（推荐，最简单）
streamSaver.mitm = 'https://jimmywarting.github.io/StreamSaver.js/mitm.html?version=2.0.0'

// 方法2：自托管（生产环境建议）
// 把 node_modules/streamsaver/mitm.html 复制到 public 目录
streamSaver.mitm = '/mitm.html'
```

**基础用法**

```js
// StreamSaver 的核心流程
const fileStream = streamSaver.createWriteStream('file.zip')
const writer = fileStream.getWriter()

// 网络数据 -> 写入流 -> Service Worker -> 浏览器下载器 -> 磁盘
const response = await fetch('https://example.com/bigfile.zip')
const reader = response.body.getReader()

while(true) {
  const { done, value } = await reader.read()
  if(done) break
  await writer.write(value)  // 数据通过 postMessage 传给 SW
}
await writer.close()
```

**为什么需要 mitm.html？**

`mitm.html` 是 **Man-In-The-Middle**（中间人）页面，它是为了解决跨域和 Service Worker 生命周期问题：

- Service Worker 不能直接创建一个"下载链接"，需要借助一个 iframe 作为中间载体
- `mitm.html` 作为这个中间人，接收数据并触发真实下载

**优点**

- ✅ 兼容性好（支持大部分现代浏览器，包括 Firefox、Safari 较新版本）
- ✅ 内存占用低（同样流式处理）
- ✅ 可以手动控制写入速度、暂停等

**缺点**

- ❌ 需要额外部署 `mitm.html` 文件
- ❌ 依赖 Service Worker，首次加载可能稍慢
- ❌ 在 Safari 中可能需要特殊配置

### 3. Range 分片并发下载

**核心原理**

这个方案的基础是 **HTTP Range 请求**。服务器可以通过响应头 `Accept-Ranges: bytes` 告诉客户端："我支持断点续传，你可以只请求文件的一部分"。

客户端利用这个特性，把大文件切成几十个片段，同时下载：

```js
# 客户端请求
GET /bigfile.zip HTTP/1.1
Range: bytes=0-10485759   # 只请求前 10MB

# 服务器响应
HTTP/1.1 206 Partial Content
Content-Range: bytes 0-10485759/104857600  # 告诉客户端这是总文件的第 0-10MB
Content-Length: 10485760
```

**完整流程：**

1. **发送 HEAD 请求**：获取文件总大小，检查是否支持 Range
2. **计算分片**：比如 100MB 文件，每片 10MB，分成 10 片，记录每片的起始和结束位置
3. **并发下载**：同时发起多个 Range 请求（通常 3-6 个并发）
4. **收集数据**：每个请求返回 `ArrayBuffer`，按顺序存在数组 `chunks[索引] = 数据`
5. **合并**：所有分片完成后，用 `new Blob(chunks)` 合并成完整文件
6. **触发下载**：通过 `createObjectURL` 或 `a.download` 保存

```js
// 简化的分片下载
const CHUNK_SIZE = 10 * 1024 * 1024  // 10MB
const totalSize = 100 * 1024 * 1024  // 100MB
const chunks = []

// 并发下载所有分片
const downloadPromises = []
for(let start = 0; start < totalSize; start += CHUNK_SIZE) {
  const end = Math.min(start + CHUNK_SIZE - 1, totalSize - 1)
  const promise = fetch(url, {
    headers: { 'Range': `bytes=${start}-${end}` }
  }).then(res => res.arrayBuffer())
    .then(data => { chunks.push(data) })
  downloadPromises.push(promise)
}

await Promise.all(downloadPromises)
const blob = new Blob(chunks)
// 触发下载...
```

**为什么能提高速度？**

- **利用带宽**：单个 TCP 连接可能跑不满带宽，多个连接可以"塞满"带宽
- **并行处理**：同时下载多个片段，总时间 ≈ 单片段下载时间 × (总片数 / 并发数)

**举例**：100MB 文件，带宽 10MB/s，单线程需要 10 秒；5 并发下载，每片 20MB，理论上只需 2 秒（服务器端允许的情况下）

**优点**

- ✅ 下载速度快（并行下载）
- ✅ 支持断点续传（可以记录已下载的分片）
- ✅ 兼容性好（只要服务器支持 Range，所有浏览器都支持）
- ✅ 可做进度控制和失败重试

**缺点**

- ❌ **内存占用高**（所有分片最终要合并到内存，100MB 文件就要占 100MB 内存）
- ❌ 需要服务器支持 Range（不支持的话只能降级）
- ❌ 分片太多会增加 HTTP 请求开销
- ❌ 合并 Blob 时可能触发浏览器内存限制

### 4. 浏览器请求

为什么浏览器自行下载 不会占用内存？

​	**浏览器自己下载时，数据直接写入磁盘缓存，不经过 JavaScript 内存。**

​	而我之前的 await batchDownloadFiles() 会把数据先读到 JavaScript 内存里。

**浏览器的架构**

```
┌─────────────────────────────────────────────────────────┐
│                    浏览器进程                            │
│  ┌─────────────────────────────────────────────────┐   │
│  │              渲染进程（你的页面）                  │   │
│  │  ┌─────────────────────────────────────────┐    │   │
│  │  │         JavaScript 堆内存                │    │   │
│  │  │    （你的 res.data、blob 在这里）         │    │   │
│  │  └─────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │              网络进程                            │   │
│  │  ┌─────────────────────────────────────────┐    │   │
│  │  │         网络缓存 / 磁盘缓存              │    │   │
│  │  │    （浏览器直接写入，不经过 JS）          │    │   │
│  │  └─────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**关键点**：

- 浏览器是**多进程**架构
- 网络进程和渲染进程是**隔离**的
- `window.open` 让网络进程直接处理下载
- 你的 `await` 把数据拉到渲染进程的 JS 内存



```js
async downLoad(id) {
  try {
    // 1. 显示加载提示（可选）
    const loading = this.$loading({
      text: '正在准备下载...',
      spinner: 'el-icon-loading'
    })
    
    // 2. 方案A：如果需要后端处理（比如记录下载日志）
    // 先请求后端，但不下载文件
    const res = await batchDownloadFiles({ ids: id }, {
      // 只获取下载链接，不下载文件数据
      responseType: 'json'
    })
    
    if (res.data && res.data.downloadUrl) {
      // 后端返回了下载链接
      window.open(res.data.downloadUrl, '_blank')
    } else {
      // 方案B：直接使用接口地址下载
      const downloadUrl = `/api/aos-cloud-flight-operation/resultdata/batchDownLoad?ids=${id}`
      window.open(downloadUrl, '_blank')
    }
    
    loading.close()
    this.$message.success('已开始下载，请查看浏览器下载栏')
    
  } catch (error) {
    this.$message.error('下载失败')
  }
}
```
