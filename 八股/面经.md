## mouseenter和mouseover有什么区别

在JavaScript中，`mouseenter`和`mouseover`都是用于检测鼠标指针移入一个元素的事件，但它们在事件触发机制上存在一个关键性的区别，这个区别主要体-现在事件冒泡（event bubbling）上。

### 主要区别：事件冒泡

*   **`mouseover`**：会冒泡。当鼠标指针移入一个元素时，会触发该元素的`mouseover`事件。之后，该事件会继续向其父元素传播（冒泡），并依次触发父元素的`mouseover`事件，直到文档的根元素。因此，当鼠标从一个元素的子元素移入时，也会触发父元素的`mouseover`事件。

*   **`mouseenter`**：不会冒泡。 `mouseenter`事件仅在鼠标指针首次从元素外部进入该元素的可视区域时触发。当鼠标指针在该元素内部的子元素之间移动时，不会再次触发`mouseenter`事件。

**简单来说**：

| 事件属性     | `mouseover`                                           | `mouseenter`                             |
| :----------- | :---------------------------------------------------- | :--------------------------------------- |
| **事件冒泡** | 会                                                    | 不会                                     |
| **触发时机** | 鼠标移入元素或其任何子元素时都会触发。                | 仅在鼠标从元素外部首次移入该元素时触发。 |
| **性能**     | 在复杂的DOM结构中，由于频繁触发，可能会引发性能问题。 | 性能更好，因为它不会在子元素上重复触发。 |

---

### 示例解析

为了更直观地理解，可以参考以下HTML结构：

```html
<div id="outer" style="padding: 50px; background-color: lightgray;">
  外层 Div
  <div id="inner" style="padding: 50px; background-color: lightblue;">
    内层 Div
  </div>
</div>
```

**使用 `mouseover` 的情况：**

如果你为`outer`元素添加了一个`mouseover`事件监听器：

1.  当鼠标指针从外部进入`outer`元素的灰色区域时，`outer`的`mouseover`事件会触发。
2.  当鼠标指针继续移动，进入`inner`元素的蓝色区域时，首先会触发`inner`的`mouseover`事件。由于事件冒泡，接着会再次触发`outer`的`mouseover`事件。
3.  当鼠标在`inner`区域内移动时，`outer`的`mouseover`事件不会再次被触发。
4.  当鼠标从`inner`移回`outer`的灰色区域时，`outer`的`mouseover`事件会再次被触发。

这种行为在某些场景下可能会导致意料之外的多次触发，例如在实现下拉菜单时，鼠标在菜单项之间移动可能会导致主菜单意外地关闭和重新打开。

**使用 `mouseenter` 的情况：**

如果你为`outer`元素添加了一个`mouseenter`事件监听器：

1.  当鼠标指针从外部进入`outer`元素的灰色区域时，`outer`的`mouseenter`事件会触发。
2.  当鼠标指针继续移动，进入`inner`元素的蓝色区域时，`outer`的`mouseenter`事件**不会**再次触发。
3.  只有当鼠标完全离开`outer`元素的边界，然后再次进入时，`mouseenter`事件才会再次触发。

### 对应的移出事件

与`mouseenter`和`mouseover`类似，`mouseleave`和`mouseout`也存在同样关于事件冒泡的区别：

*   **`mouseout`**: 会冒泡，是`mouseover`的反向操作。
*   **`mouseleave`**: 不会冒泡，是`mouseenter`的反向操作。

### 总结与应用场景

*   当你希望一个事件处理器只在鼠标首次进入一个元素的边界时被调用，并且不希望因为鼠标在其子元素上移动而重复触发时，应该使用`mouseenter`。这在实现下拉菜单、工具提示（tooltips）等交互效果时非常有用。

*   当你需要利用事件冒泡机制，例如，希望父元素能够响应其任何子元素上的鼠标移入事件时，可以使用`mouseover`。

## aync、await实现原理

`async/await` 是 JavaScript 中用于处理异步操作的一种语法糖，它使得异步代码的编写和阅读方式更接近于同步代码。其实现原理主要建立在 **Promises** 和 **Generators** 之上，并与 JavaScript 的 **事件循环 (Event Loop)** 和 **微任务队列 (Microtask Queue)** 紧密结合。

### 核心：Generators + Promises

从根本上说，`async/await` 是对 Generators (生成器函数) 的一种封装和自动化执行。

1. **Generators (生成器) - 函数的暂停与恢复**
    *   生成器函数（以 `function*` 声明）是一种特殊的函数，它可以在执行过程中被暂停，并在之后从暂停的位置恢复执行。
    *   `yield` 关键字用于暂停生成器函数的执行，并可以返回一个值。
    *   通过调用生成器的 `.next()` 方法，可以恢复函数的执行，并且可以向函数内部传递一个值，这个值会成为上一个 `yield` 表达式的结果。

    这种“暂停/恢复”的能力是实现 `await` “等待”效果的关键。

2. **Promises - 异步操作的容器**
    *   Promise 是一个表示异步操作最终完成（或失败）及其结果值的对象。
    *   `async` 函数总是隐式或显式地返回一个 Promise。 如果函数代码中返回了一个非 Promise 的值，它会被自动包装在一个已解决 (resolved) 的 Promise 中。
    *   `await` 关键字后面通常跟着一个返回 Promise 的表达式。它会“等待”这个 Promise 的状态变为 settled（即 fulfilled 或 rejected）。

### 实现原理剖析：一个自动执行的生成器

可以把 `async/await` 的工作流程想象成这样：

1.  **`async` 函数的本质**：当一个函数被声明为 `async` 时，它在底层被转换成一个特殊的生成器函数。

2.  **`await` 的作用**：`await` 关键字的功能类似于生成器中的 `yield`。当执行到 `await somePromise` 时，`async` 函数会暂停执行，并将 `somePromise` 这个 Promise 对象 `yield` (产出) 出去。

3.  **自动执行器 (Runner)**：JavaScript 引擎内部有一个“执行器”或“运行器”的角色。这个执行器会自动处理 `async` 函数（也就是那个特殊的生成器）。
    *   当 `async` 函数 `yield` 一个 Promise 时，执行器会接管这个 Promise。
    *   执行器为该 Promise 注册 `.then()` 回调。
    *   `async` 函数的执行被挂起，但**整个 JavaScript 主线程并不会被阻塞**。 引擎可以去执行其他任务。

4.  **恢复执行**：
    *   **当 Promise 成功 (fulfilled) 时**：执行器注册的 `.then()` 回调被触发。在这个回调中，执行器会调用生成器的 `.next()` 方法，并将 Promise 的解决值 (resolved value) 作为参数传入。这样，`async` 函数就从之前暂停的地方恢复执行，并且 `await` 表达式的返回值就是那个解决值。
    *   **当 Promise 失败 (rejected) 时**：执行器会调用生成器的 `.throw()` 方法，并将 Promise 的拒绝原因 (rejection reason) 传入。这使得 `async` 函数内部的 `try...catch` 块能够捕获到这个错误。

**简化的模拟代码如下：**
```javascript
// 这是一个模拟的 `async` 函数执行器
function runner(generatorFunc) {
  const iterator = generatorFunc(); // 获取迭代器

  function handle(result) {
    // 如果生成器已经执行完毕，直接返回
    if (result.done) {
      return Promise.resolve(result.value);
    }

    // 将结果包装成 Promise
    return Promise.resolve(result.value)
      .then(res => {
        // 将 Promise 的解决值传回生成器，并继续执行
        return handle(iterator.next(res));
      })
      .catch(err => {
        // 如果出错，将错误抛入生成器
        return handle(iterator.throw(err));
      });
  }

  // 启动执行器
  try {
    return handle(iterator.next());
  } catch (ex) {
    return Promise.reject(ex);
  }
}

// 使用生成器模拟 async/await
function* myAsyncFunction() {
  try {
    const result1 = yield new Promise(resolve => setTimeout(() => resolve('结果1'), 1000));
    console.log(result1); // 输出: 结果1

    const result2 = yield new Promise(resolve => setTimeout(() => resolve('结果2'), 1000));
    console.log(result2); // 输出: 结果2
  } catch (error) {
    console.error(error);
  }
}

// 运行
runner(myAsyncFunction);
```

### 结合事件循环与微任务队列

`async/await` 的执行也离不开 JavaScript 的事件循环机制。

1.  **挂起与事件循环**：当 `async` 函数在 `await` 处挂起时，它会让出 JavaScript 主线程的控制权。事件循环可以继续处理其他任务，例如用户输入、定时器回调等，从而保持了应用的响应性。

2.  **微任务队列 (Microtask Queue)**：当 `await` 等待的 Promise 解决后，`async` 函数的后续部分并不会立即执行。相反，它会被作为一个微任务 (microtask) 添加到微任务队列中。

3.  **执行时机**：事件循环在处理任务时，会优先清空微任务队列。 这意味着在一个宏任务（如 `setTimeout` 回调）执行完毕后，但在下一个宏任务开始前，所有已就绪的微任务（包括 `async` 函数的恢复执行部分）都会被执行。这保证了 `await` 后的代码能够尽快得到执行。

### 总结

`async/await` 的实现原理可以概括为：

*   **语法层面**：它是 Promise 的语法糖，旨在用同步的方式书写异步代码。
*   **核心机制**：它是 **Generators** 的暂停/恢复能力和 **Promises** 的异步状态管理的结合。
*   **执行调度**：它依赖于 **事件循环** 来避免阻塞主线程，并通过 **微任务队列** 来安排异步函数在 Promise 解决后的恢复执行。

## 事件循环

### 起点：为什么需要事件循环？

首先要记住一个关键事实：**JavaScript 是一门单线程语言**。

这意味着在任何给定时刻，JavaScript引擎只能执行一个任务。如果有一个任务非常耗时（比如一个复杂的计算或一个网络请求），那么整个程序都会被阻塞，用户界面会卡住，无法响应任何操作。这显然是不可接受的。

为了解决这个问题，JavaScript引入了异步执行和事件循环机制。它允许主线程将耗时的操作“外包”出去，然后继续执行后续的代码，而当“外包”的操作完成后，再通过一种机制回来执行相应的回调函数。

### 事件循环的核心组件

整个系统可以想象成一个协作流程，包含以下几个关键部分：

1.  **执行栈 (Call Stack)**：一个后进先出（LIFO）的数据结构。当一个函数被调用时，它会被推入栈中；当函数执行完毕返回时，它会被弹出。所有同步代码都在这里执行。

2.  **浏览器API (Web APIs) / Node.js API**：这些是宿主环境（浏览器或Node.js）提供的API，它们不是JavaScript引擎的一部分。例如 `setTimeout`, `DOM事件`, `AJAX (XMLHttpRequest)` 等。这些API可以处理耗时的操作，并且独立于JavaScript主线程运行。

3.  **任务队列 (Task Queue / Callback Queue)**：也称为 **宏任务队列 (Macrotask Queue)**。这是一个先进先出（FIFO）的队列。当一个异步操作（如`setTimeout`的计时结束、用户点击按钮）完成时，其对应的回调函数会被放入这个队列中，等待被执行。

4.  **微任务队列 (Microtask Queue)**：这是另一个先进先出（FIFO）的队列，但它拥有**更高的优先级**。专门用来存放微任务。

5.  **事件循环 (Event Loop)**：这是整个机制的“指挥官”。它是一个持续运行的进程，其工作就是不断地检查执行栈是否为空。如果为空，它就会去检查任务队列，从中取出一个任务放入执行栈中执行。

---

### 宏任务 (Macrotasks) 与 微任务 (Microtasks)

这是理解事件循环的关键区别。任务被分为了两类：

#### 1. 宏任务 (Macrotasks / Tasks)
宏任务代表一个独立的、离散的工作单元。当一个宏任务执行时，JavaScript引擎会完全投入到这个任务中，执行完毕后，可能会进行UI渲染，然后再去执行下一个宏任务。

**常见的宏任务包括：**
*   `script` (整个脚本的初始执行)
*   `setTimeout()` 和 `setInterval()` 的回调
*   `I/O` 操作（如文件读写）
*   UI事件（如 `click`, `scroll`）
*   `requestAnimationFrame`

#### 2. 微任务 (Microtasks)
微任务通常是需要**在当前任务执行结束后、下一个任务开始前立即执行**的任务。它们通常与Promise等异步操作的后续处理有关。

**常见的微任务包括：**
*   `Promise.then()`, `Promise.catch()`, `Promise.finally()` 的回调
*   `async/await` 中 `await` 关键字后面的代码（这部分本质上是Promise.then的语法糖）
*   `queueMicrotask()`
*   `MutationObserver` 的回调

---

### 事件循环的详细执行流程

现在我们可以把所有组件串联起来，事件循环的精确算法如下：

1.  首先，执行全局的 `script` 同步代码。这本身是第一个宏任务。在执行过程中，同步代码直接在执行栈中运行。
2.  如果遇到异步API（如 `setTimeout`），则将其交给对应的 Web API 处理，并注册回调函数。
3.  如果遇到微任务（如 `Promise.then()`），则将其回调函数放入 **微任务队列**。
4.  当第一个宏任务（即主脚本）执行完毕，执行栈变为空。
5.  **关键步骤**：事件循环**首先**检查 **微任务队列**。
6.  它会**执行完所有**在微任务队列中的任务，直到微任务队列变为空。如果在执行微任务的过程中，又产生了新的微任务，那么这些新的微任务也会被添加到队列的末尾，并在当前这个轮次中一并执行完毕。
7.  当微任务队列清空后，浏览器可能会进行一次UI渲染（这是一个复杂的步骤，不保证每次都渲染，但检查点在这里）。
8.  然后，事件循环会去 **宏任务队列** 中取**一个**任务（如果队列中有的话），将其放入执行栈中执行。
9.  这个新的宏任务执行完毕后，执行栈再次变为空。
10.  **返回第5步**，再次检查微任务队列，清空它，然后进行下一轮循环。

这个过程会不断重复，因此被称为“事件循环”。

**核心要点：**
*   一次事件循环只会执行**一个**宏任务。
*   但在那一个宏任务执行完毕后，会执行**所有**已存在的微任务。

---

### 代码示例与分析

让我们用一个经典的例子来走一遍这个流程：

```javascript
console.log('script start'); // 1

setTimeout(function() {
  console.log('setTimeout'); // 5
}, 0);

Promise.resolve().then(function() {
  console.log('promise1'); // 3
}).then(function() {
  console.log('promise2'); // 4
});

console.log('script end'); // 2
```

**逐步分析：**

1.  **执行第一个宏任务 (主脚本)**
    *   `console.log('script start')` 被推入执行栈并执行。**输出: `script start`**。
    *   遇到 `setTimeout(..., 0)`。`setTimeout` 是Web API，所以它的回调函数被交给浏览器计时器。即使延迟是0ms，它也必须等待主脚本执行完毕，所以回调函数被放入 **宏任务队列**。
    *   遇到 `Promise.resolve().then(...)`。`.then()` 的回调是一个微任务，所以它被放入 **微任务队列**。
    *   `console.log('script end')` 被推入执行栈并执行。**输出: `script end`**。

2.  **主脚本 (第一个宏任务) 执行完毕，执行栈为空。**

3.  **事件循环检查微任务队列。**
    *   发现微任务队列不为空，里面有 `promise1` 的回调。
    *   取出并执行 `promise1` 的回调。**输出: `promise1`**。
    *   这个回调执行后返回一个新的Promise，并链式调用了 `.then()`，所以 `promise2` 的回调作为一个**新的微任务**被放入微任务队列。
    *   事件循环继续检查微任务队列，发现它还不为空（里面有 `promise2` 的回调）。
    *   取出并执行 `promise2` 的回调。**输出: `promise2`**。

4.  **微任务队列现在为空了。**

5.  **事件循环检查宏任务队列。**
    *   发现队列中有 `setTimeout` 的回调。
    *   取出一个（也是唯一一个）宏任务，即 `setTimeout` 的回调，放入执行栈中执行。
    *   执行 `console.log('setTimeout')`。**输出: `setTimeout`**。

6.  **所有队列都为空，事件循环继续等待新的任务。**

**最终输出顺序:**
```
script start
script end
promise1
promise2
setTimeout
```

这个例子清晰地展示了微任务 (`promise1`, `promise2`) 在当前宏任务 (`script`) 结束后、下一个宏任务 (`setTimeout`) 开始前被优先且全部执行的特性。

## 垃圾回收机制

### 1. 垃圾回收的核心原理：可达性（Reachability）

JavaScript 引擎维护一个**根（Root）** 集合，能通过根访问到的值被认为是**可达的**，需要保留；无法访问到的值就是**不可达的**，可以被回收。

**根包括**：

- 全局对象 `window` / `global`
- 当前执行函数的局部变量和参数
- 调用栈上的所有变量
- 其他引擎内部持有的引用

```js
let obj = { name: '张三' }  // ① 根 → obj → {name}，可达
let arr = [obj]            // ② 根 → arr → {name}，通过另一个路径也可达

obj = null                 // ③ 切断一条路径
// {name} 仍然通过 arr[0] 可达，不会被回收

arr = null                 // ④ 切断最后一条路径
// {name} 现在没有任何根能访问到 → 变为不可达 → 等待回收
```

------

### 2. 两种主要的垃圾回收算法

#### A. 引用计数法 (Reference Counting) —— 早期使用

这是最直观的算法：跟踪记录每个值被引用的次数。

- 当声明一个变量并将一个引用类型赋值给该变量时，引用次数 +1。
- 当这个变量指向了别的值，引用次数 -1。
- **当引用次数变为 0 时**，说明没人用它了，回收！

**致命缺点：循环引用**如果 A 引用了 B，B 也引用了 A，它们的引用计数永远是 1。即使你把它们都设为 `null`，它们依然躲在内存里不走。

```js
function createCycle() {
  let objA = {}
  let objB = {}
  
  objA.ref = objB   // objB 引用数: 2（objA + 根）
  objB.ref = objA   // objA 引用数: 2（objB + 根）
  
  // 函数执行完毕，局部变量 objA、objB 被销毁
  // 但 objA.ref 和 objB.ref 互相引用
  // 引用数：objA=1，objB=1 → 永远不会归零 → 内存泄漏
}
```

#### B. 标记-清除法 (Mark-and-Sweep) —— 现代主流

这是目前 Chrome V8 引擎等现代浏览器使用的标准算法。它分为两个阶段：

1. **标记阶段**：从“根”对象（在浏览器中是 `window`）开始，遍历所有引用的对象，打上“可达”标记。
2. **清除阶段**：垃圾回收器扫描内存，发现那些**没有被标记**的对象，直接把它们清理掉。

**优点**：完美解决了循环引用问题。如果 A 和 B 互相引用但从根部无法到达，它们都会被清理。

------

### 3. V8 引擎的分代回收（优化策略）

为了提高效率，V8 引擎将内存分成了两个区域：**新生代**和**老生代**。

```
┌─────────────────────────────────────────┐
│              新生代（Young）              │
│  ┌─────────────────────────────────┐   │
│  │ 存活时间短的对象                  │   │
│  │ 如：局部变量、临时对象            │   │       使用 Scavenge 算法 回收
│  │ 回收频率高，速度快                │   │

### 4. 为什么“闭包”会引起内存泄漏？

回到你之前问的闭包。现在你应该能从底层理解了：

```
function outer() { 
	let heavyData = new Array(10000); // 很大一块内存 
    return function inner() {    
    	console.log(heavyData.length);  
    	};
    }
let stay = outer(); // 闭包形成了
```



- **分析**：全局变量 `stay` 引用了 `inner` 函数。
- **链路**：根 (window) -> `stay` -> `inner` -> `heavyData`。
- **结果**：因为这条“引用链”是通的，`heavyData` 在垃圾回收器看来是**可达的**，所以它永远不会被回收。
- **解决**：执行 `stay = null`，切断链路，垃圾回收器下次扫描时就会把 `heavyData` 清理掉。

### 5. 常见的内存泄漏场景

1. **意外的全局变量**：无意中给 `window.xxx` 赋值，导致变量永生。
2. **未清理的定时器**：`setInterval` 没执行 `clearInterval`，里面的变量会一直锁在内存里。
3. **未解绑的事件监听**：在 DOM 节点被删除后，如果没有移除 `addEventListener`，某些浏览器可能无法回收相关内存  =>   **解决**：销毁前调用 `removeEventListener`
4. **闭包**：如上所述，不当使用且未及时释放。

## 闭包

### 1. 定义

通俗地说：**闭包 = 函数 + 函数内部能够访问的外部变量**

要形成一个闭包，通常需要满足以下三个结构：

1. **函数嵌套**：一个函数内部定义了另一个函数。
2. **引用变量**：内部函数引用了外部函数的变量。
3. **函数返回**：外部函数将内部函数作为返回值，或者将其赋值给全局变量。

```js
function outer() {
  let count = 0  // 外部变量
  
  function inner() {  // 内部函数
      count++
      console.log(count)  // 访问了外部变量
  }
  
  return inner  // 返回内部函数
}

const fn = outer()  // outer 执行完毕，按理说 count 应该被销毁
fn()  // 输出 '1' —— 闭包让 count 被保留了下来
fn()  // 输出 '2'
```

### 2. 垃圾回收机制与闭包

正常情况下，函数执行完毕后，其内部的变量会被垃圾回收（GC）回收释放内存。

但当内部函数引用了外部函数的变量时，外部函数的变量**不会被回收**，因为内部函数依然持有对它的引用。

### 3. 闭包的应用场景

### 4. 闭包的注意事项

- **内存泄漏**

  闭包会持有外部函数的变量，如果使用不当，可能导致内存无法释放。

  ```js
  // 问题代码：意外的大对象被闭包持有
  function createHeavyObject() {
    const largeData = new Array(1000000).fill('数据')  // 大对象
    const smallData = 42
    
    return function() {
      // 只用了 smallData，但 largeData 也被闭包持有，无法回收
      console.log(smallData)
    }
  }

  const fn = createHeavyObject()
  // fn 存在，largeData 就一直占用内存
  ```

  **解决方案**：及时清理不需要的引用

  ```js
  function createHeavyObject() {
    let largeData = new Array(1000000).fill('数据')
    const smallData = 42
    
    const result = function() {
      console.log(smallData)
    }
    
    // 手动解除大对象的引用
    largeData = null
    
    return result
  }
  ```

- **在循环中创建闭包**

  使用 `let` 替代 `var` 可以避免经典的循环闭包问题，因为 `let` 会为每次迭代创建独立的块级作用域。

  ```js

  for (var i = 0; i < 3; i++) {

  }

  // let 解决

  }

**（2）v-model 语法糖**

​       父用 `v-model`，子用 `modelValue` prop + `update:modelValue` 事件

</template>

<!-- Child.vue -->
<template>
  <input :value="modelValue" @input="updateValue" />
</template>

<script setup>
const props = defineProps(['modelValue'])
const emit = defineEmits(['update:modelValue'])

const updateValue = (e) => {
  emit('update:modelValue', e.target.value)
}
</script>
```

**（3）.sync 修饰符（Vue2）**

- 父：`:title.sync="title"`
- 子：`$emit('update:title', 新值)`

**（4）Template Refs**

有时父组件需要直接调用子组件的方法，可以使用 `ref`。

- **Vue 2**：通过 `this.$refs.childName` 访问。

  - **父访问子：ref**

    子标签写 `ref="xxx"`

    父通过 `this.$refs.xxx` 拿到子组件实例

  - 子访问父

    子用  `this.$parent` 访问父实例（不推荐）

    <button @click="callChildMethod">调用子组件方法</button>

  const childRef = ref()

    childRef.value.increment()
    console.log(childRef.value.count)
  }
  </script>
  ```

### 2. 跨级（祖孙）组件通信

当组件嵌套太深（如 A -> B -> C -> D），用 Props 一层层传递会非常痛苦（称为 Props Drilling）。

- **Provide / Inject**：

  - **Provide**：祖先组件定义要传递给后代的数据。
  - **Inject**：任何后代组件都可以直接“注入”并使用这些数据，无论中间隔了多少层。
  - *注意*：在 Vue 2 中，provide/inject 默认不是响应式的；在 Vue 3 中配合 `ref` 或 `reactive` 可以实现完美响应。

  ```vue
  <!-- 祖先组件 -->
  <script setup>
  import { provide, ref } from 'vue'

  const theme = ref('dark')
  const updateTheme = (newTheme) => {
    theme.value = newTheme
  }

  // 提供数据和方法
  provide('theme', theme)
  provide('updateTheme', updateTheme)
  </script>

  <!-- 后代组件（任意深度） -->
  <script setup>
  import { inject } from 'vue'

  const theme = inject('theme')
  const updateTheme = inject('updateTheme')

  const switchTheme = () => {
    updateTheme(theme.value === 'dark' ? 'light' : 'dark')
  }
  </script>
  ```

## 原型链

原型链是 JavaScript 实现**继承**的核心机制。每个对象都有一个指向其原型对象的内部链接，而原型对象本身也有自己的原型，直到某个对象的原型为 `null`，这样就形成了一条链式结构。

### 1. 核心概念：三个关键属性

- **prototype（原型对象）**：
  - **谁拥有：** 只有**函数**（Function）才有。
  - **作用：** 它是函数的一个属性，存储了希望被实例共享的方法和属性。
- **__proto__（隐式原型）**：
  - **谁拥有：** 几乎**所有对象**都有。
  - **作用：** 它是一个指针，指向创建该对象的构造函数的 `prototype`。
- **constructor（构造函数）**：
  - **谁拥有：** `prototype` 对象自带。
  - **作用：** 指回该原型对象所属的构造函数。

![77477421954](C:\Users\ADMINI~1\AppData\Local\Temp\1774774219541.png) 

```js
// 每个函数都有一个 prototype 属性（箭头函数除外）
function Person(name) {
  this.name = name
}

// 每个对象都有一个 __proto__ 属性（非标准，但实际使用广泛）
const person = new Person('张三')
console.log(person.__proto__ === Person.prototype) // true

// 函数的 prototype 也是一个对象，它也有 __proto__
console.log(Person.prototype.__proto__ === Object.prototype) // true
console.log(Object.prototype.__proto__) // null
```

**区别总结：**

- `prototype`：函数特有的属性，指向该函数的原型对象
- `__proto__`：对象特有的属性，指向该对象的原型（实际指向构造函数的 prototype）

### 2. 什么是“链”？（查找机制）

当你访问一个对象的属性（比如 `obj.name`）时，JavaScript 引擎会执行以下寻宝过程：

1. **第一步：** 在 `obj` 自身属性中查找，找到则返回。
2. **第二步：** 如果没找到，就沿着 `__proto__` 去它的原型对象（`obj.__proto__`）里找。
3. **第三步：** 如果还没找到，继续沿着原型的原型往上找。
4. **终点：** 一直找到 `Object.prototype`。如果到了 `Object.prototype.__proto__`（即 `null`）还没找到，就返回 `undefined`。

**这种由 __proto__ 层层连接起来的结构，就叫原型链。**

![型](E:\Work\qianduan\JS\JS进阶\03第三天\原型链.png)

```js
// 定义构造函数
function Animal(name) {
  this.name = name
}

// 在原型上添加方法（共享）
Animal.prototype.sayHello = function() {
  console.log(`我是${this.name}`)
}

// 创建实例
const dog = new Animal('旺财')
const cat = new Animal('咪咪')

// 关系验证
console.log(dog.__proto__ === Animal.prototype)     // true
console.log(cat.__proto__ === Animal.prototype)     // true
console.log(Animal.prototype.constructor === Animal) // true
console.log(dog.constructor === Animal)              // true（沿原型链找到）

// 实例共享原型方法
dog.sayHello() // 我是旺财
cat.sayHello() // 我是咪咪

// dog自身没有sayHello方法，JS引擎查看dog.__proto__，它指向了Animal.prototype，在Animal.prototype中找到了sayHello方法，执行它
```

### 3. 原型链的尽头在哪里？

想象你在翻一本字典找“麒麟”这个词：

1. **翻页（__proto__）**：你一页一页往后翻（沿着原型链向上找）。
2. **翻到了封底（null）**：你翻到了字典的最后一页，发现后面没纸了，这就是 **null**。它告诉你：**查找过程结束了**。
3. **合上字典（undefined）**：既然翻完了都没找到，你合上字典告诉别人：“**没这个词（undefined）**”。

------

### 4. 为什么需要原型链？

- **节省内存：** 如果不使用原型，每个实例都要创建一套完整的方法，浪费空间。放在原型上，所有实例共用一份内存。
- **实现继承：** 它是 JS 实现面向对象编程的基础，允许子类复用父类的逻辑。

## vue2 和 vue3 的响应式有什么区别

### 1. 核心技术实现

- **Vue 2：Object.defineProperty**

  Vue 2 在初始化数据时，会通过 `Object.defineProperty` 遍历 `data` 里的所有属性，并将其转化为 getter/setter。它只能拦截**属性的读写**。

- **Vue 3：Proxy**

  Vue 3 使用了 ES6 的 `Proxy` 对象来创建响应式代理。它不直接修改原对象，而是拦截**整个对象的操作**。

### 2. 核心差异对比

## v-for中的 :key中为什么使用id，而不是index

在 Vue 的 `v-for` 循环中，`:key` 的存在不是为了给开发者看，而是为了给 **Vue 的虚拟 DOM 算法（Diff 算法）** 看的。

简单来说，`key` 是节点的 **“身份证号”**。当数据变化时，Vue 通过这个身份证号来判断哪些元素是新生的、哪些是老熟人、哪些该被销毁。

------

### 1. 核心区别：稳定性

| **特性**     | **使用 ID (唯一标识)**                 | **使用 Index (索引)**                       |
| ------------ | -------------------------------------- | ------------------------------------------- |
| **来源**     | 数据库自带或后端生成的唯一字段         | 当前数组的下标（0, 1, 2...）                |
| **唯一性**   | 在全局或同级中永久唯一                 | 随位置变化而变化                            |
| **性能表现** | **极高**，精确复用 DOM 节点            | **一般**，频繁增删时会导致大量重新渲染      |
| **安全性**   | 状态保持正确（如下面提到的输入框内容） | 容易导致状态错乱（如 Input 内容、勾选框等） |

------

### 2. 深入理解：为什么 ID 比 Index 好？

我们可以通过 **“增删”** 场景来直观感受两者的差异。

### 场景 A：在列表头部插入数据

假设你有一个列表 `[A, B, C]`，现在你在开头插入一个 `D`。

- **使用 Index 作为 Key：**
  1. 旧：`{0:A, 1:B, 2:C}`
  2. 新：`{0:D, 1:A, 2:B, 3:C}`
  3. **结果：** Vue 发现 `key: 0` 对应的东西变了（从 A 变成了 D），它会认为这一行被“修改”了。紧接着它发现 1、2、3 全变了。Vue 会**强制更新或重新渲染**几乎所有节点，效率极其低下。
- **使用 ID 作为 Key：**
  1. 旧：`{id1:A, id2:B, id3:C}`
  2. 新：`{id4:D, id1:A, id2:B, id3:C}`
  3. **结果：** Vue 发现 `id1, id2, id3` 都在，只是位置挪了。它只需要**移动**现有的 DOM 节点，并新创建一个 `id4` 即可。

------

### 3. Index 可能引发的“灾难”：状态错乱

这是前端面试中最常考的知识点。如果你的列表中包含**临时状态**（比如 `<input>` 输入的内容、复选框的勾选状态），使用 Index 会产生 Bug。

**例子：**

1. 你在列表第 1 行（Index 0）的输入框里写了“我要买苹果”。
2. 你在列表最前面插入了一个新行。
3. **Bug 出现了：** 原本属于第一行的“我要买苹果”，现在居然出现在了新插入的那一行里。

**原因：** Vue 的“就地复用”机制。Vue 看到 `key: 0` 还在，就直接复用了旧的 DOM 节点。虽然数据（Props）变了，但输入框里的 HTML 状态（Value）还没变，导致信息“串位”。

------

### 4. 什么时候可以用 Index？

虽然 ID 是“标准答案”，但在以下极端情况下，用 Index 也是安全的：

1. **静态列表：** 列表仅仅是展示，永远不会有增删、排序、过滤等操作。
2. **数据极少：** 且没有表单输入、子组件等复杂状态。
3. **没有 ID：** 如果后端确实没给 ID，且数据也不涉及上述 Bug 风险。
4. **纯尾部操作**：只push、pop，不涉及头部、中间操作

## 数据类型

### **1. 基本类型**

**string, number, boolean, null, undefined, symbol, bigint**

**核心特性：**

（1）**存储在栈 (Stack) 中**：因为它们占用空间固定且小。

（2）**不可变性 (Immutability)**：你不能修改字符串的某个字母。当你“修改”一个字符串时，JS 其实是销毁了旧的，创建了一个全新的。

（3）**按值传递**：赋值时直接复印一份，互不影响

**null、undefined区别：**

![77484115455](C:\Users\ADMINI~1\AppData\Local\Temp\1774841154559.png)

### **2. 引用类型**

**object, array, function, date, regexp, map, set 等**

**核心特性：**

（1）**存储在堆 (Heap) 中**：因为它们的大小不固定（比如数组可以无限 push），栈内存里只存一个指向堆的**地址**（钥匙）。

（2）**可变性 (Mutability)**：你可以随时增删对象的属性或数组的元素。

（3）**按引用传递**：赋值时，复制的是那把“钥匙”。

### 3. 总结：如何准确判断它们？

在前端开发中，仅靠 `typeof` 是不够的（因为它分不清数组和对象，还会把 null 认错）。

- **typeof**：适合判断基本类型（除了 `null`）。

  ![77484103286](C:\Users\ADMINI~1\AppData\Local\Temp\1774841032864.png)

- **instanceof**：适合判断具体的引用类型（如 `arr instanceof Array`）。

- **终极方案**：`Object.prototype.toString.call(value)`

  - 这个方法能精确返回 `[object Array]`、`[object Null]`、`[object Date]` 等。

## 深拷贝和浅拷贝

### 1. 浅拷贝

浅拷贝只复制对象的第一层属性。如果属性是基本类型，复制的就是**值**；如果属性是引用类型（对象或数组），复制的就是**内存地址**。

```js
// 基本类型 - 直接赋值就是拷贝值
let a = 10
let b = a
b = 20
console.log(a) // 10 ✅ 互不影响

// 引用类型 - 直接赋值只是拷贝引用
let obj1 = { name: '张三' }
let obj2 = obj1
obj2.name = '李四'
console.log(obj1.name) // '李四' ❌ 相互影响
```

- **后果：** 对于引用类型，拷贝后的新对象和原对象**共享**内部的子对象。修改其中一个，另一个也会跟着变。

- **常用方法：**

  - `Object.assign()`
  - 展开运算符 `{...obj}`
  - `Array.prototype.slice()`

  ```js
  const original = {
    name: '张三',
    age: 18,
    address: {
      city: '北京',
      district: '朝阳区'
    },
    hobbies: ['读书', '跑步']
  }

  const shallowCopy = { ...original }

  // 修改第一层属性 - 不影响原对象
  shallowCopy.name = '李四'
  shallowCopy.age = 20
  console.log(original.name)  // '张三' ✅
  console.log(original.age)   // 18 ✅

  // 修改第二层引用类型 - 影响原对象
  shallowCopy.address.city = '上海'
  shallowCopy.hobbies.push('游泳')
  console.log(original.address.city)  // '上海' ❌
  console.log(original.hobbies)       // ['读书', '跑步', '游泳'] ❌
  ```

- **手动实现浅拷贝**

  ```js
  function shallowClone(obj) {
    if (typeof obj !== 'object' || obj === null) return obj
    
    const newObj = Array.isArray(obj) ? [] : {}
    
    for (let key in obj) {
      // 检查一个属性是否属于对象“自己”，而不是从“原型链”上继承来的
      if (obj.hasOwnProperty(key)) {
        newObj[key] = obj[key]  // 直接赋值
      }
    }
    
    return newObj
  }

  // 测试
  const original = {
    a: 1,
    b: { c: 2 }
  }
  const copy = shallowClone(original)
  copy.b.c = 3
  console.log(original.b.c)  // 3 ❌
  ```

### 2. 深拷贝 

深拷贝会递归地复制对象的所有层级。它会在堆内存中开辟一块全新的空间，把原对象的所有内容（包括嵌套的子对象）完整地克隆一份。

- **后果：** 两个对象完全**独立**，互不影响。修改新对象不会对原对象产生任何副作用。

- **常用方法：**

  - **简单粗暴法：** `JSON.parse(JSON.stringify(obj))`
    - *缺点：* 
      - 无法复制函数、undefined、Symbol
      - 无法复制 Date、RegExp、Map、Set 等特殊对象
      - 无法处理循环引用
      - 会忽略原型链上的属性


  - **现代标准法：** `structuredClone(obj)` (原生支持，推荐)
  - **库函数：** Lodash 的 `_.cloneDeep()`

  ```js
  const original = {
    name: '张三',
    age: 18,
    address: {
      city: '北京',
      district: '朝阳区'
    },
    hobbies: ['读书', '跑步'],
    sayHi: function() {
      console.log('你好')
    },
    date: new Date(),
    reg: /test/g,
    undefined: undefined,
    null: null,
    symbol: Symbol('test')
  }

  const deepCopy = JSON.parse(JSON.stringify(original))

  // 验证完全独立
  deepCopy.address.city = '上海'
  deepCopy.hobbies.push('游泳')

  console.log(original.address.city)  // '北京' ✅
  console.log(original.hobbies)       // ['读书', '跑步'] ✅

  // ❌ JSON 方法的局限性：
  console.log(deepCopy.sayHi)         // undefined - 函数丢失
  console.log(deepCopy.date)          // 字符串，不是 Date 对象
  console.log(deepCopy.reg)           // {} - 正则表达式丢失
  console.log(deepCopy.undefined)     // 丢失
  console.log(deepCopy.symbol)        // 丢失
  console.log(deepCopy)               // 不包含函数、undefined、symbol
  ```

#### 正则表达式的拷贝

正则表达式包含**模式**（pattern）和**标志**（flags），需要完整复制这两部分。

```js
// 正则的组成部分
const regex = /abc/gi
console.log(regex.source)      // 'abc'（source：正则表达式的模式文本）
console.log(regex.flags)       // 'gi'（flags：标志字符串）
console.log(regex.global)      // true
console.log(regex.ignoreCase)  // true
console.log(regex.multiline)   // false

// 正则的构造函数形式
const regex2 = new RegExp('abc', 'gi')
console.log(regex2.source)  // 'abc'
console.log(regex2.flags)   // 'gi'
```



```js
// 方法1：使用 source 和 flags
function copyRegex1(regex) {
  return new RegExp(regex.source, regex.flags)
}

// 方法2：使用构造函数（直接传递正则对象）
function copyRegex2(regex) {
  return new RegExp(regex)  // ES6+ 支持
}

// 方法3：手动提取标志
function copyRegex3(regex) {
  const flags = []
  if (regex.global) flags.push('g')
  if (regex.ignoreCase) flags.push('i')
  if (regex.multiline) flags.push('m')
  if (regex.dotAll) flags.push('s')
  if (regex.unicode) flags.push('u')
  if (regex.sticky) flags.push('y')
  
  return new RegExp(regex.source, flags.join(''))
}

// 方法4：使用 toString() 解析（不推荐，但了解原理）
function copyRegex4(regex) {
  const str = regex.toString()  // '/abc/gi'
  const lastSlash = str.lastIndexOf('/')
  const pattern = str.slice(1, lastSlash)
  const flags = str.slice(lastSlash + 1)
  return new RegExp(pattern, flags)
}

// 测试
const original = /abc/gi
const copy1 = copyRegex1(original)
const copy2 = copyRegex2(original)

console.log(copy1.source === original.source)  // true
console.log(copy1.flags === original.flags)    // true
console.log(copy1 === original)                // false（不同对象）
```

### 3. 形象的比喻

想象你有一张**藏宝图**（对象），地图上标记了一个**宝箱的位置**（子对象）。

- **浅拷贝：** 你复印了一张地图。现在有两张地图，但它们指向的是**同一个宝箱**。如果有人去把宝箱里的金子换成了石头，两张地图的使用者都会发现宝箱变了。
- **深拷贝：** 你不仅复印了地图，还**克隆了一个一模一样的宝箱**放在另一个地方。现在你有两张地图和两个独立的宝箱，你在其中一个箱子里放火，另一个箱子安然无恙。

### 4.场景对比表

| **特性**     | **浅拷贝**             | **深拷贝**           |
| ------------ | ---------------------- | -------------------- |
| **复制深度** | 仅限第一层             | 无限层级（递归）     |
| **内存空间** | 与原对象共享子对象地址 | 开辟完全独立的新空间 |
| **性能**     | 快                     | 慢（尤其是大对象）   |
| **修改影响** | 修改嵌套对象会相互影响 | 完全互不影响         |
